# Languages Feature

The Languages feature allows you to manage a catalog of languages in the system. It provides full CRUD (Create, Read, Update, Delete) operations.

## Overview

Languages are the foundation for the greetings feature. Each language has:
- A unique identifier (UUID)
- A human-readable name (e.g., "English")
- A language code (e.g., "en")
- Timestamps for creation and updates

## Use Cases

- Add new languages to support additional greetings
- Update language information
- List all available languages
- Retrieve specific language details
- Remove languages (will cascade delete associated greetings)

## Data Model

```csharp
public record Language
{
    public Guid Id { get; init; }
    public string Name { get; init; }
    public string Code { get; init; }
    public DateTime CreatedAt { get; init; }
    public DateTime UpdatedAt { get; init; }
}
```

## Example Usage

### Create a Language
```bash
curl -X POST http://localhost:8080/api/languages \
  -H "Content-Type: application/json" \
  -d '{
    "name": "French",
    "code": "fr"
  }'
```

Response:
```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",

  "code": "fr",
  "createdAt": "2024-01-15T10:30:00Z",
  "updatedAt": "2024-01-15T10:30:00Z"
```

## Implementation Details

The Languages feature is implemented using:
- **Repository Pattern** - `LanguageRepository` handles all data access
- **Dapper** - Lightweight ORM for efficient SQL queries
- **Minimal APIs** - Clean, functional endpoint definitions

### Logging
All operations are logged with structured logging:
```csharp
logger.LogInformation("Creating language with code {LanguageCode}", request.Code);
logger.LogInformation("Updated language with id {LanguageId}", id);
```

### Validation
Basic validation ensures:
- Name and Code are not empty
- Database constraints enforce uniqueness
